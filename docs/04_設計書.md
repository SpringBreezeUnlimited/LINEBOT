# 設計書

## 1. 設計方針
- 単一の Flask アプリで Web 管理と LINE 予約受付を統合する。
- 責務を main.py に集約しつつ、テンプレートと静的ファイルで画面処理を分離する。
- DB を状態の単一ソースとし、予約状態の遷移を管理する。

## 2. アーキテクチャ
### 2.1 構成要素
- Web サーバー: Flask + Gunicorn
- 永続化: PostgreSQL
- 外部連携: LINE Messaging API
- 表示層: Jinja2 + CSS + JavaScript

### 2.2 処理の流れ
1. ユーザーが LINE でメッセージ送信
2. LINE が Webhook を `main.py` に転送
3. `process_reservation` が受付内容を判定
4. DB 更新と LINE 応答を実施
5. 管理者は Web 画面で状態変更
6. 管理画面 JS が `/admin/data` を使って一覧を更新

## 3. レイヤー設計
### 3.1 プレゼンテーション層
- `templates/` の HTML
- `static/css/` と `static/js/`

### 3.2 アプリケーション層
- `main.py` の各ルート関数
- 認証、入力検証、状態遷移、通知制御

### 3.3 永続化層
- PostgreSQL テーブル
- 設定値、予約、種類、監査ログ、レート制限記録を保持

## 4. 主要モジュール責務
### 4.1 認証・セキュリティ
- ログイン認証
- セッション管理
- CSRF 検証
- Host/HTTPS 制御
- レート制限
- 監査管理者向けの管理者アカウント管理

### 4.2 予約処理
- 予約受付
- キャンセル
- 到着受付
- 呼出と完了

### 4.3 管理画面処理
- 一覧表示
- 種類管理
- 履歴表示
- CSV 出力
- ログ表示

### 4.4 自動呼出処理
- 設定人数の自動呼出
- 呼出タイムアウト時の自動キャンセル
- 呼出結果の保存

### 4.5 待ち時間算出処理
- 待機人数ベースの目安待ち時間算出
- 予約時・待ち時間照会時の利用者通知
- 集計値の設定テーブル保存

## 5. 状態遷移
- `waiting` → `called`
- `called` → `arrived`
- `arrived` → `done`
- `waiting` / `called` / `arrived` → `cancelled`

補足:
- `called` は一定時間経過でタイムアウトし `cancelled` へ遷移する。

## 7. パフォーマンス設計
- 初期描画はサーバーサイド HTML を基本とする
- 管理画面の追加更新は `/admin/data` で差分取得する
- 初期 JSON の埋め込みは行わない
- JS は起動時の処理を最小化し、必要時にのみ更新する

## 8. 保守設計
- 設定値は環境変数で変更できる
- DB スキーマは初回実行時に必要分だけ整備する
- バージョンは `APP_VERSION` で画面表示する

## 9. API 設計メモ
- 管理画面向け JSON: `/admin/data`, `/admin/type_counts`
- バッチ向け JSON: `/tasks/process-call-queue`
- 監査管理者向け操作: `/admin/admin-accounts`, `/admin/admin-accounts/<account_id>/login-id`
